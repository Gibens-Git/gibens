2. FIXED: UNREAD BADGE NOT CLEARING (TAB BADGE)
   Root cause #1: markJobNotificationsRead was missing type='new_message'
     filter and used a fragile JSONB filter on data->>job_id.
   Root cause #2: Supabase JS v2 query builders are LAZY — the HTTP
     request only fires when .then() or await is called. All Supabase
     calls that were fire-and-forget (no .then(), no await) silently
     never executed.
   Fix: replaced markJobNotificationsRead with markAllMessageNotifsRead
     (clears all new_message notifications for the user), added
     .then(() => {}) to force execution in both Chat.tsx files.
   Files changed:
     gibens/packages/supabase/src/queries.ts — added markAllMessageNotifsRead
     gibens/apps/pro/src/pages/Chat.tsx — use markAllMessageNotifsRead
     gibens/apps/customer/src/pages/Chat.tsx — use markAllMessageNotifsRead
   Commits: 016d226, 566aaba

3. FIXED: THREAD-LEVEL UNREAD BADGE (MESSAGE LIST "2" BADGE)
   Root cause: markMessagesRead (which calls the mark_messages_read RPC)
     was also fire-and-forget — messages.is_read was never updated in
     the DB. get_customer_chat_list and get_vendor_chat_list both compute
     unread_count from messages.is_read = FALSE, so the badge persisted.
   Fix: added .then(() => {}) to markMessagesRead in both Chat.tsx files.
   Files changed:
     gibens/apps/pro/src/pages/Chat.tsx
     gibens/apps/customer/src/pages/Chat.tsx
   Commit: 3de8d38

   KEY LESSON: Any Supabase query builder call that is not awaited or
   .then()'d will silently not execute. Always add .then(() => {}) or
   await to fire-and-forget Supabase calls.

4. FIXED: PHOTO UPLOAD FAILING
   Root cause: job-photos storage bucket did not exist in Supabase.
   Fix steps:
     a) Created bucket "job-photos" in Supabase Storage UI
     b) Added storage RLS policies via SQL Editor:
          CREATE POLICY "Authenticated users can upload job photos"
          ON storage.objects FOR INSERT TO authenticated
          WITH CHECK (bucket_id = 'job-photos');

          CREATE POLICY "Public can read job photos"
          ON storage.objects FOR SELECT TO public
          USING (bucket_id = 'job-photos');
     c) Toggled bucket to "Public bucket" in bucket settings
        (required for getPublicUrl() to serve images without auth)
   Added try/catch error handling to customer Chat.tsx handlePhoto
   so failures show in the UI instead of silently doing nothing.
   Files changed:
     gibens/apps/customer/src/pages/Chat.tsx — error handling in handlePhoto
   Commit: cf54b84

   STATUS: Customer can send photos, pro receives them and they display.
   Photo upload on the PRO side (pro sending to customer) not yet tested.
