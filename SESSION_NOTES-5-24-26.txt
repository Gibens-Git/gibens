WHAT WAS DONE THIS SESSION
=======================================================

1. FIXED: PRO RLS ERROR WHEN SENDING MESSAGES
   Root cause: messages_insert policy did EXISTS(SELECT FROM jobs) which
   went through jobs_select RLS. jobs_select only showed vendors jobs in
   'open'/'bidding' status. After bid acceptance job becomes 'accepted',
   making it invisible in the subquery → INSERT denied.
   Migrations applied:
     009_fix_jobs_select_and_messages.sql — rebuilt jobs_select so vendors
       can see any job they have a bid on regardless of status; added
       SECURITY DEFINER to mark_messages_read (no UPDATE policy on messages)
     010_fix_messages_rls_direct.sql — rewrote messages_read/insert to use
       direct bids check instead of jobs subquery
     011_messages_security_definer.sql — final/definitive fix: added
       user_can_message_job() SECURITY DEFINER helper, rebuilt
       messages_read and messages_insert to use it, added
       get_vendor_chat_list() and get_customer_chat_list() SECURITY
       DEFINER RPCs for the Messages list pages
   Files changed:
     supabase/migrations/009_fix_jobs_select_and_messages.sql (new)
     supabase/migrations/010_fix_messages_rls_direct.sql (new)
     supabase/migrations/011_messages_security_definer.sql (new)
     gibens/apps/pro/src/pages/Messages.tsx — now uses get_vendor_chat_list RPC
     gibens/apps/customer/src/pages/Messages.tsx — now uses get_customer_chat_list RPC

2. FIXED: MESSAGES LIST PAGE SHOWING "NO MESSAGES YET"
   Root cause: pro Messages page used a nested bids->jobs->messages query.
   Each join went through a different RLS policy; the chain silently
   filtered out messages for accepted jobs.
   Fix: replaced with SECURITY DEFINER RPCs (see migration 011 above).
   Messages list now works correctly.

3. FIXED (PARTIAL): UNREAD BADGE NOT CLEARING
   Root cause #1 (fixed): badge counted ALL unread notifications (bid
     accepted, job confirmed, etc.), not just new_message type.
     Fix: useNotifications now filters by type='new_message' everywhere.
   Root cause #2 (fixed): markJobNotificationsRead was not called in the
     realtime subscription callback — only on initial load and polling.
     If a message arrived via realtime, its notification stayed unread.
     Fix: added markJobNotificationsRead call to realtime callback in
     both Chat.tsx files.
   Files changed:
     gibens/apps/pro/src/hooks/useNotifications.ts
     gibens/apps/customer/src/hooks/useNotifications.ts
     gibens/apps/pro/src/pages/Chat.tsx
     gibens/apps/customer/src/pages/Chat.tsx
